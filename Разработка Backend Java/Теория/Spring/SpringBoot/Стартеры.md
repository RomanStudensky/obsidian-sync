3Стартер - это зависимость, которая подтягивает зависимости по назначению и делает автоконфигурацию этих зависимостей - "Подключил и работает".

Стартер почти не содержит код: он тянет нужные библиотеки и модуль `*-autoconfigure`, который регистрирует бины по условиям.

# Как встраивается в приложение

 Добавляем `implementation("org.springframework.boot:spring-boot-starter-xxx")`.
- При старте Spring Boot подхватывает классы из модуля `xxx-autoconfigure`:
    - `@AutoConfiguration` классы,
    - условия `@ConditionalOnClass`, `@ConditionalOnMissingBean`, `@ConditionalOnProperty`,
    - настройки через `@ConfigurationProperties`.
- С Boot 3 автоконфигурации перечисляются в файле  
    `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`.

# Чем стартер отличается от библиотеки

- **Библиотека** — предоставляет код/классы/API. Её нужно **настраивать вручную**.
- **Стартер** — это **обертка над зависимостями** + (часто) **автоконфигурация**, которая сама создаёт бины и подключается через условия.
- Итого: библиотека = _код_, стартер = _комфорт подключения библиотеки + авто-настройка

# Когда писать стартер, а когда библиотеку

- **Стартер**:
    - нужно “plug-and-play”, умолчания, внешние свойства (`application.yml`);
    - интеграция с Boot (health, metrics, tracing, etc.).
- **Библиотека**:
    - общий утилитарный код без привязки к Spring;
    - узкоспециализированный модуль, который владелец приложения конфигурирует сам.
- Часто делают **2 модуля**: `my-lib` (код) + `my-lib-autoconfigure` (автоконфиг) и **3-й тонкий**: `spring-boot-starter-my-lib` (зависимость на первые два).
