Тип индекса в реляционных базах данных, который определяет физический порядок хранения строк в таблице на основе значений ключа индекса. Иными словами, данные в таблице хранятся отсортированными по ключу кластерного индекса.

**Ключевые особенности кластерного индекса:**

- **Физический порядок данных:** Строки таблицы упорядочены на диске в соответствии с ключом индекса, что ускоряет доступ к данным при выполнении запросов, использующих этот ключ.
- **Единственность:** В одной таблице может быть только один кластерный индекс, поскольку физический порядок данных может быть только один.
- **Листовые страницы содержат данные:** в структуре индекса листовые узлы содержат сами данные таблицы, а не указатели на них.

**Пример работы кластерного индекса:**

Представим таблицу **employees** с полем **employee_id**, по которому создаётся кластерный индекс.

- **Создание кластерного индекса:**
```sql
CREATE CLUSTERED INDEX idx_employee_id ON employees(employee_id);
```

- **Результат:**

- Данные в таблице **employees** на диске будут отсортированы по возрастанию значений **employee_id**.
- Запросы типа SELECT * FROM employees WHERE employee_id = 123 будут выполняться быстрее, так как система может быстро найти нужные данные благодаря упорядоченности.

**Особенности в различных СУБД:**

- **Microsoft SQL Server:**

- Поддерживает кластерные индексы.
- При создании первичного ключа по умолчанию создаётся кластерный индекс.

- **MySQL (InnoDB):**

- Первичный ключ автоматически становится кластерным индексом.
- Если первичный ключ не определён, InnoDB создаст скрытый кластерный индекс.

- **PostgreSQL:**

- Не имеет нативной поддержки кластерных индексов в том же смысле, что и SQL Server или MySQL.
- Команда CLUSTER позволяет переупорядочить данные таблицы в соответствии с указанным индексом, но это одноразовая операция.
- Для поддержания порядка после вставки новых данных команду CLUSTER нужно выполнять периодически.

**Преимущества кластерных индексов:**

- **Быстрый доступ к данным:** Улучшают производительность запросов, которые возвращают диапазоны значений по ключу индекса.
- **Эффективное использование дискового пространства:** так как данные и индекс хранятся вместе.

**Недостатки кластерных индексов:**

- **Медленные операции вставки и обновления:** Поддержание физического порядка данных требует дополнительных ресурсов при изменении данных.
- **Фрагментация:** Частые изменения могут привести к фрагментации данных, требуя периодической реорганизации.

**Заключение:**

Кластерные индексы эффективны для ускорения чтения данных по определённому ключу, но могут замедлить операции записи. Их использование должно быть обосновано профилем нагрузки на базу данных и характером выполняемых запросов.
